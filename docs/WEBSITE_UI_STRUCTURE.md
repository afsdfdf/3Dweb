# 整个网站 UI 全结构说明

## 1. 网站 UI 总体定位

本项目的网站 UI 不是单一的营销官网，而是一个由多个界面层组成的产品型站点。整体可分为四个前台可感知层级：

1. **品牌与营销层**
   - 负责展示产品价值、能力说明、案例、价格、资源与转化入口。
   - 主要页面：`/`、`/features`、`/solutions`、`/showcase`、`/pricing`、`/resources`

2. **生成入口层（Studio Entry）**
   - 负责把用户从“了解产品”带到“开始创建”。
   - 主要页面：`/generate`

3. **结果与交付层**
   - 负责展示生成状态、结果预览、下载与打印承接。
   - 主要页面：`/results/[taskCode]`

4. **用户工作台层（Dashboard）**
   - 负责管理任务、模型、订单、积分与账号。
   - 主要页面：`/dashboard` 及子页面

整个网站 UI 的本质是：
**营销站负责转化，Studio 负责创建，结果页负责承接，Dashboard 负责管理。**

---

## 2. 全站 UI 框架结构

## 2.1 站点公共壳层

全站前台主要通过 `SiteShell` 作为统一外壳，承担以下 UI 结构职责：

### 顶部区域
- **公告栏**
  - 用于放促销、公告、系统提示或产品信息。
- **主导航**
  - 承载营销页导航，如功能、方案、案例、价格、资源等。
- **品牌区**
  - Logo / 产品名 / 品牌标识。
- **用户操作区**
  - 未登录：登录、注册
  - 已登录：工作台入口、退出登录、用户身份标签

### 产品导轨区域
- 用于快速跳转到核心产品区域：
  - `Studio`
  - `Dashboard`
  - `任务`
  - `模型`
  - `订单`
  - `API`
  - `Admin`

### 主内容区域
- 根据当前页面承载不同内容模块。

### 页脚区域
- 展示品牌补充信息、方向说明、产品说明。

## 3. 页面结构分层

## 3.1 首页 `/`

首页目前是极简入口型首页，整体结构如下：

### A. Hero 介绍区
- 品牌眉题
- 主标题
- 一段简短说明文字

### B. 双入口卡片区
- 图生 3D 入口卡片
- 文生 3D 入口卡片
- 每张卡片包含：
  - 类型标签
  - 标题
  - 简短描述
  - 进入按钮
  - 视觉装饰区域

### C. 作品预览区
- 区块标题
- “查看更多”入口
- 作品卡片列表
- 每张作品卡包含：
  - 分类
  - 标题
  - 摘要
  - 装饰视觉预览

### 首页 UI 角色
- 首页不是完整内容门户，而是一个“快速进入产品”的入口页。
- 当前结构偏轻，适合快速启动，但若用于商业转化，建议后续补充更多内容层。

---

## 3.2 营销内容页

包括：
- `/features`
- `/solutions`
- `/showcase`
- `/pricing`
- `/resources`
- `/developers`

这些页面复用 `MarketingPage` 组件，结构基本一致：

### A. 紧凑型 Hero 区
- 页眉题
- 页面标题
- 页面说明
- 主 CTA
- 次 CTA（可选）

### B. 内容分区网格
每个 section 通常包含：
- section 标题
- section 说明
- 卡片组或要点列表

### C. 卡片型内容块
用于展示：
- 功能说明
- 方案价值
- 资源内容
- 开发者能力
- 定价信息

### 营销页 UI 角色
- 负责“解释产品”和“支撑转化”。
- 页面结构统一，利于内容复用与 CMS 驱动。

---

## 3.3 登录页 `/login`

### A. 引导文案区
- Login 眉题
- 页面主标题
- 简短说明文案

### B. 登录表单区
- 表单主体
- 输入项
- 提交按钮

### 登录页 UI 角色
- 从公开站点进入个人工作流的认证入口。
- 视觉应尽量简洁，避免过多干扰元素。

---

## 3.4 注册页 `/register`

结构与登录页基本一致：

### A. 引导文案区
- Register 眉题
- 页面主标题
- 注册后价值说明

### B. 注册表单区
- 注册信息输入
- 提交操作

### 注册页 UI 角色
- 承接首页与营销页 CTA。
- 负责把访客转化成可进入 Studio / Dashboard 的用户。

---

## 3.5 生成页 `/generate`

这是网站最核心的创建页面之一，当前结构可视为“任务创建台”。

### A. 页面顶区
- 页面标题
- 状态标签（如上传中 / 可提交）

### B. 生成模式选择区
- 图生 3D
- 文生 3D
- 图文混合

每张模式卡片包含：
- 单选控件
- 模式名称
- 模式说明

### C. 参数配置区
- 风格选择
- 输出格式选择

### D. 质量预设区
- 标准
- 高质量
- 超高质量

### E. 输入内容区
- 上传参考图
- 图片预览
- 文本描述输入框

### F. 提交操作区
- 预计积分消耗
- 提交按钮
- 错误信息反馈

### 生成页 UI 角色
- 从用户视角完成“创建任务”的核心流程。
- 是从营销页进入产品能力的关键转化页。

---

## 3.6 结果页 `/results/[taskCode]`

这是任务创建后的结果承接页，结构相对完整，承担“状态展示 + 结果预览 + 交付动作”三种职责。

### A. 顶部概览区
- 结果页眉题
- 任务编号
- 提示词描述
- 当前状态
- 当前进度

### B. 结果主内容区

#### 1. 模型预览区
- 模型查看器
- HUD 叠层信息
- 可用格式信息
- 视图模式按钮

#### 2. 任务状态说明区
- 当前任务状态
- 进度条
- 输入模式
- 供应商
- 预扣积分
- 创建时间 / 更新时间

#### 3. 结果摘要卡片
- 模型状态
- 结果承接说明

### C. 右侧行动区

#### 1. 下一步动作卡
- 下载 GLB
- 下载 STL
- 创建打印订单
- 打开模型库

如果未完成：
- 展示处理中说明
- 展示轮询与等待提示

#### 2. 交付信息卡
- 可用格式
- 打印状态
- 任务编号
- 模型名称

#### 3. 交付建议卡
- 下一步使用建议
- 下载 / 模型库 / 打印流程说明

### 结果页 UI 角色
- 是生成链路的“承接页”和“动作页”。
- 用户从这里决定下一步进入下载、归档还是打印流程。

## 5. 整站 UI 信息流

从用户使用路径看，整站 UI 可以抽象为以下信息流：

### 路径 1：访客转化路径
`首页 / 营销页 -> 注册 / 登录 -> 进入生成页`

### 路径 2：创建路径
`生成页 -> 提交任务 -> 结果页`

### 路径 3：资产沉淀路径
`结果页 -> 模型库 -> 下载 / 归档`

### 路径 4：交付路径
`结果页 / 模型库 -> 创建订单 -> 订单中心`

### 路径 5：长期管理路径
`工作台总览 -> 任务 / 模型 / 订单 / 积分 / 设置`

这说明网站 UI 不是孤立页面集合，而是完整业务流的界面编排。

## 7. 当前整站 UI 的结构特征

### 优点
- 已有完整的产品路径：营销 -> 创建 -> 结果 -> 管理
- 壳层统一，导航与页面风格具备一致性
- 工作台结构清晰，用户业务对象划分明确
- 结果页已经具备较强的产品承接能力

### 当前偏向
- 首页偏“快速入口”
- Dashboard 偏“业务工作台”
- 结果页偏“承接交付页”
- 营销页偏“统一模板页”

### 适合后续加强的方向
- 首页增加更完整的产品信息层
- 营销页增加更强视觉层次和案例感
- 生成页改为更明显的步骤型 UI
- 结果页强化阶段状态与下一步引导
- Dashboard 首页继续强化任务优先级提示

---

## 8. 一句话总结

整个网站 UI 结构已经形成了较完整的产品骨架：

**公开页面负责讲清价值，生成页负责开始任务，结果页负责承接交付，Dashboard 负责长期管理。**

后续若继续优化，应围绕这条主线增强页面层级、视觉节奏和用户路径引导，而不是把所有页面都做成同一种“卡片展示页”。
