🗄️ 数据层与 Payload Collections

评分 68 · 最高风险区
🔴 严重 积分账本「双写 + 三处真相」一致性风险 已核验
位置 src/lib/creditLedger.ts:366-391 · src/hooks/syncCreditBalanceMirror.ts:24 · src/collections/Credits.ts:23
问题 账本事务用原生 SQL 直接 UPDATE credits 与 UPDATE users.credits_balance(绕过 Payload hooks);而 Credits.afterChange 又经 req.payload.update 把余额镜像写回 users。同一字段 users.credits_balance 存在三条写路径(原生 SQL、镜像 hook、Admin 手改),原生路径不触发镜像 hook。
影响 余额为冗余反范式字段,任一路径漂移即导致账实不符(资金问题),且难以排查。
建议 以 credits.balance 为唯一真相,users.credits_balance 改为只读派生;所有写入统一收敛到同一服务层。
🔴 严重 账本事务与 Payload 请求事务相互独立,跨操作无原子性 已核验
位置 src/lib/ledgerStore.ts:22-51 · src/lib/creditLedger.ts:274
问题 withLedgerTransaction 从 req.payload.db.drizzle.$client 自取连接开 BEGIN/COMMIT,与 Payload 自身 req.transactionID 事务并非同一事务。
影响 同一业务流程若既调账本原生事务、又调 payload.create/update,二者无法共同回滚 —— 出现「扣费成功但任务落库失败」的孤儿扣款。
建议 复用 Payload 事务(透传 req 与同一 drizzle transaction),或把业务写入纳入同一原生事务。
🟠 高 Supabase 与 Payload 双 Schema 并存,数据源混乱 已核验
位置 src/lib/supabase/queries.ts:233,265 · src/lib/supabase/billing.ts:23,56
问题 查询层仍读写旧 Supabase 业务表 public.profiles / ai_tasks / credit_ledger_entries / models,而 Payload 实际拥有 users / generation-tasks / credit_transactions / models(主键类型亦不同:UUID vs integer)。billing.ts:56 甚至 UPDATE public.profiles.stripe_customer_id,与 Users.stripeCustomerId 是两份数据。
影响 结构性隐患;supabase/reset-public-business-schema.sql 已 drop 这些表,线上若未建则相关查询全部失败。
建议 明确以 Payload 为唯一业务数据源,下线或改写 supabase/queries.ts、billing.ts 的旧表读写。
🟠 高 破坏性迁移删除本地化表,但配置实际未用本地化
位置 src/migrations/20260529_020000_posts_default_content_locale.ts:133-134
问题 DROP TABLE posts_locales / _posts_v_locales,而 Posts.ts 无任何 localized:true 字段。down 回滚仅恢复 en 行,其他 locale 数据永久丢失,实质不可逆。

建议 确认产品不再需要 Posts 多语言;迁移头部显著标注数据丢失风险。
🟠 高 计数字段无 DB 层并发保护,易计数漂移
位置 Models.ts:155-209(view/likes/favorites/comments Count)· Users.ts:257-277(followers/following)

建议 计数走原生 UPDATE ... SET likes_count = likes_count + 1,或依赖已有唯一索引(model_likes_user_model_unique_idx)做幂等后聚合。

位置 CreditTransactions.ts:25-26,45-46 · PrintOrders.ts:24-26 · GenerationTasks.ts:24,82 · Models.ts:46-47
